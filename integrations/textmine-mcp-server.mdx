The **TextMine MCP server** is the TextMine-hosted Streamable HTTP MCP endpoint for external agent clients. It lets Codex, Claude, Cursor, Windsurf, ChatGPT/OpenAI, Copilot Studio, and automation clients discover and call TextMine tools backed by Public API V3.

Use it when an external agent needs to search, read, or automate TextMine work. Use [agent self-service signup](/integrations/third-party-agent-signup) when that agent needs its own TextMine organization, service-account API key, and billing account.

## How it fits

TextMine has three agent-facing surfaces that are easy to confuse:

| Surface | What it is for |
| --- | --- |
| **TextMine MCP server** | Hosted TextMine product tools for external MCP clients. It is backed by Public API V3 and uses TextMine API keys. |
| **ReadMe MCP endpoint** | Developer-hub MCP surface for API reference discovery and generated API operations from [developers.textmine.com](https://developers.textmine.com). |
| **Workbench MCP providers** | In-app provider browsing and import flows that let Workbench use connected systems as temporary chat context. |

For product semantics, stay in this docs site. For exact operation IDs, schemas, and client configuration snippets, use [API and developer docs](/integrations/api-and-developer-docs) and the developer hub.

## What it exposes

The MCP server is implemented as a thin wrapper over Public API V3. Each MCP tool should map to an existing V3 endpoint rather than adding new business logic in the MCP service.

Current tool areas include:

- Vaults and Vault documents.
- Document search, create, update, and delete.
- Reports and report templates.
- Workflows and workflow catalogs.
- Agents and long-running task-agent work.
- Alerts.
- Team and Vault users.
- Business records and record schemas where enabled.
- Integrations, provider connection status, provider folder listing, and provider file listing.

When an agent needs exact request or response shapes, route it to the [Public API reference](https://developers.textmine.com/reference). When it needs product meaning, route it back to Mintlify.

## Authentication model

Direct MCP clients should call the server with a TextMine API key:

```text
Authorization: Bearer tm_<key>
```

Platforms that own the `Authorization` header can use the alternate API-key header:

```text
X-TextMine-Api-Key: tm_<key>
```

The MCP server normalizes either form before forwarding the request to Public API V3. Public API resolves the key to the real service-account or user actor, organization, and permissions.

For external agents that need a dedicated organization and service-account API key in the multitenant, non-enterprise deployment, use [agent self-service signup](/integrations/third-party-agent-signup). Enterprise and dedicated deployments should contact TextMine support.

## MCP registry metadata

TextMine keeps a registry metadata draft in the product repository as `back-end/services/mcp/server.json`. The official MCP Registry remote-server format uses a `server.json` file with a `remotes` entry, `type: "streamable-http"`, the remote MCP URL, and required auth headers. See the official [MCP remote server publishing guide](https://modelcontextprotocol.io/registry/remote-servers).

Before listing TextMine MCP in external MCP catalogs:

1. Confirm the production MCP endpoint is publicly reachable and stable.
2. Confirm whether the listed endpoint should be `https://app.textmine.com/mcp` or an environment-specific host.
3. Confirm the registry name and ownership verification strategy.
4. Validate `server.json` with the current registry schema or publisher CLI.
5. Publish to the official MCP Registry first, then submit the same remote endpoint to community directories if the launch plan calls for broader discovery.

## Import and provider support

TextMine MCP clients can use provider-facing tools where the connected provider supports them. That includes listing integrations, checking provider connection status, listing provider folders or files, and uploading a provider file into Vault through TextMine.

Use this split when documenting or building agents:

- Import into **Workbench** when the file is temporary context for chat, comparison, summarization, or drafting.
- Import into **Vault** when the file should be processed, extracted, reviewed, governed, used by workflows, or published onward to Legislate.
- Use **Public API V3** when an external system or agent needs a durable integration point.

## Agent rules

- Prefer named, narrow tools over general-purpose actions.
- Read before mutating when the current state matters.
- Ask for confirmation before creating, deleting, publishing, sending, assigning, or triggering workflow actions.
- Use idempotency keys for retryable create or trigger operations when the API supports them.
- Cite document, Vault, workflow, run, record, or task identifiers in agent output so humans can audit the action later.
- Keep `tm_...` API keys out of prompts, tickets, screenshots, source code, and shared docs.
