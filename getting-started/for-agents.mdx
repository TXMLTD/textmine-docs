## Product routing rules

- Use [**Vault**](/vault/overview) for active document work: upload, import, processing, document types, extracted fields, tags, Ask Vault, reports, reprocessing, and workflow triggers.
- Use [**Legislate**](/legislate/overview) for validated team documents that have been published from Vault.
- Use [**Workbench**](/workbench/overview) for exploratory multi-step work across TextMine capabilities.
- Use [**Workflows**](/workflows/overview) when the user wants the same process to run again, run from a trigger, or produce a durable run history.
- Use [**Playbooks**](/playbooks/overview) for reusable document review guidance.
- Use [**Records**](/records/overview) when document evidence should become structured business data.
- Use [**Scribe**](/scribe/overview) when the output is a generated document or template.

Do not describe [Vault](/vault/overview) as [Legislate](/legislate/overview), or Legislate as Vault. A document can be processed in Vault and later published to Legislate, but those are separate surfaces.

## MCP surfaces

TextMine has three agent-related surfaces that sound similar but serve different jobs.

| Surface | What it is for |
| --- | --- |
| TextMine MCP server | TextMine's own hosted MCP service for external agent clients. It exposes TextMine tools backed by Public API V3. |
| ReadMe MCP endpoint | The developer hub MCP endpoint generated from the API reference at `developers.textmine.com/mcp`. Use it for API discovery and execution from the docs. |
| Workbench MCP providers | In-app provider connections that let Workbench browse and import external files into a Workbench chat. |

Use the [TextMine MCP server](/integrations/textmine-mcp-server) when the agent client should discover first-class TextMine tools. Use Public API V3 when the agent needs exact endpoint control or a system-to-system integration.

## Agent self-service signup

Use [agent self-service signup](/integrations/third-party-agent-signup) when an external agent or automation platform needs its own organization, service-account API key, billing account, and MCP/Public API access in the multitenant non-enterprise deployment.

Autonomous clients should discover signup metadata first:

```text
GET /.well-known/textmine-agent.json
GET /v3/third-party-agents/signup/discovery
GET /v3/third-party-agents/signup/availability
```

The trusted runtime should decrypt and store the returned `tm_...` API key outside model-visible context. The model should only see non-secret identifiers such as `credentialRef` and `apiKeyPrefix`.

## Safe execution rules

Agents should not invent IDs or assume access. Discover first, then act.

For API-backed work, use the developer hub operation IDs and follow this default order:

1. Discover accessible Vaults, users, providers, document types, workflow catalogs, record schemas, agents, or report templates.
2. Confirm the target resource with the user if names are ambiguous.
3. Ask before destructive actions, credential creation, scheduled agents, enabled workflows, provider disconnects, broad imports, or billing checkout.
4. Poll asynchronous work after upload, provider import, reprocessing, task-agent work, report generation, and workflow runs.
5. Keep bearer tokens out of prompts, screenshots, source code, and shared docs.
6. Include document, Vault, workflow, run, record, or task identifiers in user-facing summaries.

## When to send agents to developer docs

Send agents to [developers.textmine.com](https://developers.textmine.com) when they need:

- Public API V3 operation IDs.
- Request and response schemas.
- MCP setup for Codex, Claude, Cursor, Windsurf, or ChatGPT/OpenAI.
- Agent playbooks for provider import, direct upload, reports, workflows, metadata validation, records, or exports.

Stay in Mintlify when the task is about product behavior, feature naming, user workflows, screenshots, or explaining how TextMine surfaces relate to each other.
